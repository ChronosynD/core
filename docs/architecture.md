# Architecture

ChronosynD is a pipeline. Kernel-side BPF programs emit raw syscall events. The userspace collector normalizes them. The feature extractor turns disjoint per-process windows into fixed-shape feature vectors. The scoring engine compares each vector against the cached baseline for that process. The tamper-evident store persists baselines and an audit log of every state-changing operation.

```
 kernel                            userspace
 ------                            -----------------------------------------------
                                   ┌───────────┐    ┌──────────┐    ┌──────────┐
 [syscall] ─ BPF probe ─ ringbuf ─▶│ collector │ ─▶ │ features │ ─▶ │ scoring  │ ─▶ alerts
                                   └───────────┘    └──────────┘    └──────────┘
                                                                          ▲
                                                                          │ warm cache
                                                                          │
                                                                ┌─────────┴─────────┐
                                                                │ storage           │
                                                                │  SQLite + audit   │
                                                                │  hash chain       │
                                                                └───────────────────┘
                                                                          ▲
                                                                          │
                                                                  Sediment / Naive
                                                                  baseline fits
```

The dataflow is one-directional through the runtime path. The store is read at startup to warm the scorer's in-memory cache and written when a fitting pass produces a new baseline or an operator tags a maintenance window through the CLI.

## Crate responsibilities

| Crate | Responsibility | Status |
|---|---|---|
| `chronosynd-bpf` | In-kernel programs in restricted C, plus the build-pipeline scaffolding | Source written, compiled into the collector via `--features bpf` on Linux |
| `chronosynd-collector` | EventSource trait, synthetic event source for testing, BPF-backed source for Linux, daemon main loop, JSONL recording and replay helpers | Daemon runs end to end including BPF, with `--record` capturing replayable traces |
| `chronosynd-features` | `SyscallNgramExtractor` over disjoint per-process windows, feature vector contract shared with Python | Implemented and tested |
| `chronosynd-baseline` | `Baseline` trait, the shared `IndependentGaussianBaseline` plumbing, `NaiveBaseline`, and `Sediment` with the bias-corrected trimmed estimator | Implemented and parity-checked |
| `chronosynd-scoring` | In-memory scorer with per-process thresholds, plus a `warm_from_store` helper that bridges from the storage layer | Implemented and tested |
| `chronosynd-storage` | SQLite-backed baseline persistence with a SHA-256 hash chain over a write-ahead audit log of every state-changing operation | Implemented and tested, with direct tamper-injection cases |
| `chronosynd-cli` | `chronosynd baseline list/show`, `chronosynd maintenance start/end/current`, `chronosynd verify-store`, `chronosynd fit-baseline`, `chronosynd fit-from-trace` | Implemented as a library plus a thin binary, integration-tested |

## Research and production split

`chronosynd-py` is the canonical home of every algorithm. Research iteration happens there because Python with NumPy and SciPy is the right tool for fitting estimators, sweeping hyperparameters, and producing paper figures. Once an algorithm is frozen on the Python side, the Rust port lands in the corresponding crate, and CI starts enforcing bit-equivalent outputs across a fixed set of test vectors generated by the Python implementation.

Concretely:

- `chronosynd_py.parity.emit` generates 15 deterministic baseline test cases at three sample sizes and four estimator configurations, with 20 score inputs per case.
- The Rust integration test at [`chronosynd-rs/crates/baseline/tests/parity_tests.rs`](../chronosynd-rs/crates/baseline/tests/parity_tests.rs) loads the same JSON file and asserts every score matches within 1e-9 absolute or relative tolerance.
- `chronosynd_py.parity.emit_features` generates 4 deterministic event-stream cases that exercise the syscall n-gram extractor across multiple window sizes, vocabularies, and process keys.
- The Rust integration test at [`chronosynd-rs/crates/features/tests/parity_tests.rs`](../chronosynd-rs/crates/features/tests/parity_tests.rs) loads that file and asserts every emitted feature vector matches the Python reference within 1e-12 tolerance.
- Both checks run from [`scripts/check_parity.sh`](../scripts/check_parity.sh) and the [`parity` GitHub workflow](../.github/workflows/parity.yml).

## Tamper-evident store

The threat model assumes the adversary cannot corrupt the on-disk baseline store undetected. The storage layer enforces this with a SHA-256 hash chain over the audit log:

- Every `put_baseline`, `start_maintenance_window`, and `end_maintenance_window` runs in a single SQLite transaction that commits both the data row and an audit row atomically.
- Each audit row's hash covers `(seq, ts_ns, operation, length-prefixed payload, prev_hash)`.
- The chain is anchored at the all-zero hash, so the first row's `prev_hash` field is a fixed sentinel.
- `BaselineStore::verify_audit_log()` walks the chain start to end, recomputes every hash, and reports the first sequence number where the recomputed value disagrees with the stored value, or marks the log clean.

Three direct tamper-injection tests in [`chronosynd-rs/crates/storage/tests/store_tests.rs`](../chronosynd-rs/crates/storage/tests/store_tests.rs) cover three classes of attack: payload modification, hash forgery, and row deletion. All three are detected at the row where the chain breaks. The CLI exposes this through `chronosynd verify-store`, which exits with code 2 on tamper detection so an operator's monitoring system can tell it apart from a CLI crash.

## Daemon main loop

The daemon binary in [`chronosynd-collector`](../chronosynd-rs/crates/collector/src/daemon.rs) wires every component together. At startup it opens the store, optionally seeds a demo baseline, warms the scorer's cache via `warm_from_store`, and constructs a `SyscallNgramExtractor` with the default Linux syscall vocabulary. It then drains events from an `EventSource`, feeds each event into the extractor, and scores any emitted feature vectors against the cached baselines. Alerts print with an `[ALERT]` prefix. Below-threshold emissions print as `[ok]`. Cache misses print as `[err]`.

The synthetic event source is the demo path used by the integration tests and lets the daemon run and be tested cross-platform. The BPF-backed source lives behind the same `EventSource` trait under `cfg(all(target_os = "linux", feature = "bpf"))` and is wired in by passing `--bpf` at runtime.

## Record and replay

The collector can serialize every observed event to a JSONL file in addition to scoring it live. Passing `--record <path>` opens an append-mode writer that runs in front of the feature extractor: each parsed `Event` is written as one JSON line before being fed into the n-gram window. The wire format is in [`chronosynd-rs/crates/collector/src/wire.rs`](../chronosynd-rs/crates/collector/src/wire.rs) and uses a flat `WireEvent` struct with the kind stored as a raw kernel code, so unknown future codes round-trip without breaking older readers.

The CLI's `fit-from-trace` subcommand inverts the flow. It reads a recording, filters events to the requested `process_key`, replays them through a fresh `SyscallNgramExtractor`, and feeds the emitted feature vectors to the chosen estimator. Because the extractor is the canonical one used at runtime, baselines fit from a recording are interchangeable with baselines fit from a live event stream. The whole record-replay path is exercised by [`chronosynd-rs/crates/collector/tests/record_replay_tests.rs`](../chronosynd-rs/crates/collector/tests/record_replay_tests.rs) and [`chronosynd-rs/crates/cli/tests/cli_tests.rs`](../chronosynd-rs/crates/cli/tests/cli_tests.rs).

## Why these abstractions

Two design decisions worth calling out.

The feature extractor takes primitives (`process_key: &str, syscall_nr: u32`) rather than borrowing the collector's `Event` struct. This avoids a circular dependency between `chronosynd-features` and `chronosynd-collector`, and it keeps the features crate independent of how events are represented upstream, whether from a real BPF source, a synthetic source, or a replayed trace.

The scorer does not own the storage layer. It holds an in-memory cache of `StoredBaseline` records and exposes `upsert_baseline`, `forget`, and `score`. A separate `sync::warm_from_store` helper reads the store once at startup and pushes baselines into the scorer. This keeps the scoring path pure CPU and free of I/O dependencies, and it makes the scorer trivially testable without touching SQLite.
